# HYDRA Sprint 2A — Self-Generating Detection Engine
## Claude Code Prompt

---

You are building Sprint 2A for HYDRA, an AI-powered SOC investigation automation platform. This sprint activates Layer 3 of the moat: the self-generating detection engine. HYDRA observes attack patterns across investigations and automatically generates Sigma detection rules that can be pushed to customer SIEMs. This is the revenue expansion flywheel — investigation → detection → customer value → more customers.

## Context

By this sprint, HYDRA has:
- 50K+ investigations in the corpus (production + bootstrap)
- Entity graph with cross-tenant intelligence (Sprint 1K)
- Model tiering with prompt versioning (Sprint 1I)
- LLM call logging for future fine-tuning

The investigation corpus contains structured data: attack techniques, entity relationships, risk scores, verdicts, analyst feedback. Patterns emerge: certain entity-technique-edge combinations repeat across investigations. These patterns can be automatically expressed as Sigma detection rules.

## Deliverables

### 2A-1: Attack Pattern Mining

Create `worker/detection/pattern_miner.py`:

A Temporal workflow that runs daily and identifies repeating attack patterns:

1. **Query investigation corpus for patterns:**
```sql
-- Find technique-entity combinations that appear in 5+ true_positive investigations
SELECT 
    unnest(i.attack_techniques) as technique,
    eo.role,
    e.entity_type,
    ee.edge_type,
    COUNT(DISTINCT i.id) as investigation_count,
    AVG(i.risk_score) as avg_risk_score,
    COUNT(DISTINCT i.tenant_id) as tenant_spread
FROM investigations i
JOIN entity_observations eo ON eo.investigation_id = i.id
JOIN entities e ON e.id = eo.entity_id
LEFT JOIN entity_edges ee ON (ee.source_entity_id = e.id OR ee.target_entity_id = e.id)
    AND ee.investigation_id = i.id
WHERE i.verdict IN ('true_positive', 'suspicious')
    AND i.source = 'production'
GROUP BY technique, eo.role, e.entity_type, ee.edge_type
HAVING COUNT(DISTINCT i.id) >= 5
ORDER BY investigation_count DESC;
```

2. **Cluster patterns into detection candidates:**
   - Group by ATT&CK technique
   - Require: minimum 5 investigations, minimum 2 tenants, average risk_score >= 50
   - Extract: typical entity types involved, edge patterns, timing characteristics

3. **Store detection candidates:**
```sql
CREATE TABLE IF NOT EXISTS detection_candidates (
    id UUID DEFAULT gen_random_uuid() PRIMARY KEY,
    technique_id VARCHAR(20) NOT NULL,
    pattern_signature TEXT NOT NULL,  -- hash of the pattern for dedup
    entity_types TEXT[],
    edge_patterns TEXT[],
    investigation_count INTEGER,
    tenant_spread INTEGER,
    avg_risk_score FLOAT,
    status VARCHAR(20) DEFAULT 'candidate' CHECK (status IN (
        'candidate', 'generating', 'validating', 'approved', 'deployed', 'retired'
    )),
    sigma_rule TEXT,        -- generated Sigma YAML
    validation_result JSONB, -- test results
    created_at TIMESTAMPTZ DEFAULT NOW(),
    deployed_at TIMESTAMPTZ,
    UNIQUE(pattern_signature)
);
```

### 2A-2: Sigma Rule Generator

Create `worker/detection/sigma_generator.py`:

For each detection candidate, generate a Sigma rule using the LLM:

1. **Build context for the LLM:**
   - Pattern description: technique, entity types, edge patterns
   - Example investigation summaries (3-5 from the corpus that match this pattern)
   - MITRE ATT&CK technique description (from mitre_techniques table)

2. **Generate Sigma rule:**
   - Use the "reasoning" model tier for quality
   - System prompt: "You are a detection engineer. Generate a Sigma rule that detects the following attack pattern."
   - Enforce structured output: valid YAML that conforms to Sigma spec
   - Include: title, description, status, level, logsource, detection logic, falsepositives, tags (ATT&CK technique IDs)

3. **Sigma rule template (ensure LLM outputs conform):**
```yaml
title: "HYDRA-{technique_id}: {description}"
id: {uuid}
status: experimental
level: {high|medium|low}
description: |
    Auto-generated by HYDRA from {investigation_count} investigations
    across {tenant_spread} environments.
author: HYDRA Detection Engine
date: {YYYY/MM/DD}
tags:
    - attack.{tactic}
    - attack.{technique_id}
logsource:
    category: {category}
    product: {product}
detection:
    selection:
        {field}: {value}
    condition: selection
falsepositives:
    - {known false positive scenarios}
```

4. **Validate generated rule:**
   - Parse as YAML — must be valid
   - Check required Sigma fields present
   - Check logsource category is valid
   - Check detection logic has selection + condition

### 2A-3: Rule Validation Pipeline

Create `worker/detection/rule_validator.py`:

Before a rule is deployed, test it:

1. **Syntax validation:** Parse Sigma YAML, verify structure
2. **Historical testing:** Run the rule logic against investigation corpus
   - True positive rate: does it match investigations where this technique was confirmed?
   - False positive rate: does it match investigations that were benign/false_positive?
   - Require: TP rate >= 70%, FP rate <= 20%
3. **Store validation results in detection_candidates.validation_result**
4. **Auto-approve if:** TP rate >= 80% and FP rate <= 10%
5. **Flag for review if:** TP rate 70-80% or FP rate 10-20%
6. **Reject if:** TP rate < 70% or FP rate > 20%

### 2A-4: Rule Registry + Versioning

Create `worker/detection/registry.py`:

```sql
CREATE TABLE IF NOT EXISTS detection_rules (
    id UUID DEFAULT gen_random_uuid() PRIMARY KEY,
    candidate_id UUID REFERENCES detection_candidates(id),
    technique_id VARCHAR(20) NOT NULL,
    rule_name VARCHAR(255) NOT NULL,
    rule_version INTEGER NOT NULL DEFAULT 1,
    sigma_yaml TEXT NOT NULL,
    status VARCHAR(20) DEFAULT 'active' CHECK (status IN ('active', 'testing', 'retired')),
    tp_rate FLOAT,
    fp_rate FLOAT,
    investigations_matched INTEGER,
    tenant_spread INTEGER,
    created_at TIMESTAMPTZ DEFAULT NOW(),
    retired_at TIMESTAMPTZ,
    UNIQUE(technique_id, rule_version)
);

CREATE INDEX idx_rules_technique ON detection_rules(technique_id);
CREATE INDEX idx_rules_status ON detection_rules(status);
```

### 2A-5: Detection API Endpoints

Add to Go API:

`GET /api/v1/detections/rules`
- Returns all active Sigma rules for the requesting tenant
- Filterable by technique_id, status, level
- Returns Sigma YAML that can be directly imported into SIEMs

`GET /api/v1/detections/rules/{id}/sigma`
- Returns raw Sigma YAML for a specific rule
- Content-Type: application/x-yaml

`GET /api/v1/detections/stats`
- Total rules, rules by status, rules by technique, recent additions

### 2A-6: Skill Factory Integration

After a new detection rule is approved:

1. Check if a corresponding HYDRA investigation skill exists for that technique
2. If the rule caught patterns the skill missed → flag skill for improvement
3. Store skill performance data: which skills produce true_positives vs false_positives
4. This is the seed of the Skill Factory — automated skill quality tracking

## Important Constraints

- Generated rules must NEVER contain tenant-specific data (IPs, usernames, internal hostnames)
- Rule generation uses the "reasoning" model tier — this is a quality-critical task
- All generated rules are tagged with `author: HYDRA Detection Engine` and include generation metadata
- Air-gap mode: rules are stored locally, no push to external SIEMs (that's Sprint 2C)
- The pattern mining query must be efficient with 50K+ investigations — use proper indexes, consider materialized views
- Sigma rules must conform to the Sigma specification: https://github.com/SigmaHQ/sigma-specification
